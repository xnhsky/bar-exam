## 4. 執筆規約の要点（詳細は §v13p）

- **配置**：物語側＝該当ラベル段落 `<p data-fa-label…>` の直後に兄弟 `<div>`／カード側＝`.syn-path` 閉じ直後・🗝記憶のフック（`.syn-image`）の前。
- **data-dgm 同期**：`{番号}-{連番}`（例 `391-01`）。1論点1枚を執筆し、物語＋対応カードへ**同一文字列**で複製（G67 が不一致を ERROR）。論点が複数記述にまたがる時は複数カードへ同じ id で置いてよい（380-04＝カード4・5、383-02＝カード2・4・5 の実績）。
- **verdict 色の意味論（corpus で統一済み）**：
  - `is-ok` 緑=成立・あり・当たる ○／`is-ng` 赤=不成立・なし・当たらない ✕
  - `is-teal`/`is-acc`=中立対比（どちらも法・罪名や処理が違うだけ。例：偽造/変造、吸収/牽連）
  - `is-acc`「必要」/`is-flat`「不問・不要」（要件の要否対比。例：行使の目的、券面の外観）
  - 図解の ○✕ は**法的帰結**の層。記述の正誤（G60 命題の真偽）とは別層なので混同しない。
- **記号フリー**：甲乙・A・①等の問題都合ラベルを使わない（「本人」「係員」「実在団体」等の実体語に開く）。
- **タイトル**：口語で対比の軸を一言（「── 」区切り。例「偽造か変造か ── 新しい権利を生んだか、中身をいじっただけか」）。
- **出典**：`<span class="dgm-src">（最判昭27.12.25）</span>` を rule 末尾に。**表記はそのファイル本文の形式に合わせる**。判/決や年月日に確信がなければ載せない（§6）。
- **内容の出所**：図解はそのファイルの PATH/TRAP/物語に**既にある**規範・判例の再構成が原則。ファイルに無い新規の法的主張を足さない。

## 5. アンカーの実務（ここが唯一の罠どころ）

- アンカーは**生 HTML の一意な部分文字列**。判例名が `<a class="ref-case"…>` で包まれていると平文アンカーは外れる→リンクタグごと含める。
- カード側 syn-path の閉じは**ファイルで3形状**を確認済み：
  1. `…</span></p> </div>`（同一行・380/383/385 型）
  2. `…</span></p>\n</div>`（閉じが別行・381 型）→ アンカーに `\n</div>` まで含める（insert が CRLF へ自動変換）
  3. `…</span></p>      </div>`（6スペース・386-390 型）
  実物は `grep` か extract 後に該当文末を `repr` で確認してから spec を書く。
- **同一アンカーに2枚積む場合**（例：390 場面5 とカード4）：後から挿入した方が上に入るため、**視覚順 A,B にしたければ spec には B,A の順**で書く。

## 6. 品質規律（Opus 運用で特に）

- 機械ゲート PASS＝構造 OK の意味。**執筆者本人の自己照合1回**（結論の向き・条文番号・レーンとカード本文の一致）は省かない。
- **省エネ規律 (c)**：確信の持てない判例（非著名・下級審・判/決や日付が怪しい）だけ Web 一次確認。**Opus では Fable 比でここを固く**。リモートは裁判所サイト等が proxy 403 になることがある→解決できない時は 379 方式（出典を図解に載せない／本文は触らずローカル確認事項として報告に残す）。
- 図解の対比を組む過程で**既存解説の矛盾を見つけたら最小差分で是正**し、コミットメッセージに明記（実績：386 カードウ「有価証券→通貨」、387 カードオ「148条→152条」）。
