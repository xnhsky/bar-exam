# 正典改定 → 既存アップデート プレイブック（再発防止）

> TX/JX/RX/ARIADNE のいずれでも、**正典（canonical スケルトン／spec）を改定し、既存生成物を
> 新版へ揃え直す**ときに同じ事故が繰り返し起こる。本書はその原因と恒久対処をまとめる。
> 実例：TX360 インライン正典化で codex に 刑TX366〜385 を「更新」させたところ、20 問すべてが
> 接ぎ木（旧エンジン流用＋後付けパッチ）になり、何度修正依頼しても収束しなかった
> （顛末と修復は `docs/fix-tx-lex-recanon.md`）。

---

## 1. なぜ起こるのか（4 つの根本原因）

### 原因① 「既存を手で更新」は AI に“接ぎ木”を強制する
「この既存ファイルを新デザインに直して」という指示は、AI に **§7 禁止の「保守的書き換え」**
を強いる。AI は旧構造を温存したまま新要素を**上から足し**、足りない挙動を**ランタイムの
後付けパッチ**で埋める。結果＝旧エンジン＋新マークアップ＋band-aid script の三層接ぎ木。
- 実例：旧 Annex C JS（407行）を残し、インラインカードを `tx-inline-v1211-upgrade-js` で後付け。
- 症状：デザイン崩れ・一部機能しない・本文肥大（+30〜70K）・「いらないものが入る」。

### 原因② 検証が「存在（presence）」しか見ていない
既存の検証（TX G系／JX J系／RX R系）は主に「必要要素が**在るか**」を確認する。
**土台（CSS/エンジン）の載せ替え漏れ・二重実装・禁止パッチ・本文肥大**は検出できない。
→ **検証 ALL PASS なのに見た目・機能が壊れている**という最悪の見逃しが起きる。

### 原因③ LLM の自由編集は非決定論的で、修正するほど発散する
同じ指示でも出力が割れる（RX で CSS が 58 種に割れた前例と同型）。修正依頼を重ねると
**パッチにパッチ**を当て続け、根本（旧ベース流用）が消えないまま分岐が増える＝収束しない。

### 原因④ 改定が複数層に及ぶと手作業で全層一致は不可能
正典改定は通常 **構造シェル＋CSS＋JSエンジン＋本文契約（属性・初期状態）** の複数層に及ぶ。
人手／LLM 手編集で全層を canonical と一致させるのは現実的に無理。1 層直すと別層がずれる。

## 2. 恒久対処（改定のたびに必ずやる 5 点セット）

### 対処A 既存移行は「LLM 手編集」ではなく「決定論的 recanon スクリプト」で
内容を保持したまま**土台だけ canonical からバイト一致で流し込む**専用スクリプトを書く。
本文（問題固有内容）は一切触らない。これが唯一収束する方法。
- 一族：`rx-recanon.py`／`rx-restyle.py`／`rx-bodyfix.py`／`tx-lex-recanon.py`。
- 設計原則：①canonical から `<style>`/`<script>` を採用（AI 選定パレット等の固有値だけ保存）
  ②禁止パッチ・旧エンジンを物理削除 ③本文契約（属性・初期状態）を機械正規化 ④**冪等**
  ⑤既に正典・対象外のファイルは触らない。

### 対処B 改定と同時に「整合（identity）ガード」検証を 1 つ足す
presence ガードに加え、**「canonical エンジン/CSS と一致するか」「禁止パッチが無いか」
「script 本数」**等の整合チェックを新設する。これが「PASS なのに壊れている」を塞ぐ。
- 実例：`validate-tx-core.py` の **G41**（canonical エンジン固有関数の必須化＋band-aid 禁止＋
  script 本数上限）。正実装 360-365 は PASS／接ぎ木 366-385 は全 FAIL に分離。
- 判別子の選び方：**正実装と接ぎ木で確実に差が出る固有シグネチャ**（固有関数名・固有 id・
  本数）を使う。中身の質ではなく構造の同型で機械判定する。

### 対処C その整合ガードを push 前ゲートに常駐させる
1 ファイル検証は呼び忘れる。**配布前ゲート**（`check-lexia-preflight.py`）に組み込み、
リモート（`jx-push.sh`）・ローカル（`jx-finalize.ps1`）の両動線で commit/push 前に自動で弾く。
- 実例：`check-tx-lex-engine.py` を preflight ステップに追加（接ぎ木を含む push を停止）。

### 対処D 「正実装 gold」を必ず数問残し、基準にする
改定時、正典クローン動線で作った**正しい実装を 1〜数問**確保する（例：刑TX360〜365）。
移行スクリプトの移植元・整合ガードの判別基準・目視の比較対象すべてがこれを基準にできる。
**gold が無いまま一括移行を始めない。**

### 対処E 改定ごとに「移行プレイブック」を 1 枚残す
誰が・何を・どのコマンドで・どのゲートで確認するかを `docs/fix-*.md` に書く。
LLM／codex に渡す指示は**「HTML を開かず・スクリプト実行→検証→push」**に固定する。

## 3. 改定時の標準手順（チェックリスト）

正典（canonical/spec）を改定するときは、改定 PR / 作業に必ず以下を含める：

- [ ] **gold を作る**：新正典クローン動線で正実装を 1〜数問生成し検証通過（移行の基準）。
- [ ] **整合ガードを追加**：presence だけでなく「canonical と同型か／禁止パターンが無いか」を
      機械検出する検査を `validate-*.py` に足す（gold が PASS・旧/接ぎ木が FAIL になることを実証）。
- [ ] **push 前ゲートに組込**：その検査を `check-lexia-preflight.py`（または相当ゲート）に常駐。
- [ ] **決定論的移行スクリプト**：既存一括移行は recanon/restyle 系を書く（内容保持・冪等・対象限定）。
      LLM 手編集での一括更新はしない。
- [ ] **移行プレイブック**：`docs/fix-*.md` に手順（実行→検証→push・手編集禁止）を残す。
- [ ] **dry-run で対象確認 → --apply → 全数検証 → gold と目視比較**の順で適用。
- [ ] PDF から作り直すしかない問（本文構造が想定外でスクリプトが対象にできない）は
      手で繕わず `/new-tx`・`/new-jx` 等で**再生成**する。
- [ ] **付帯属性を落とさない（判例百選 `data-hyakusen`）**：TX `_lex`／ARIADNE の case-card 開始タグには
      Lexia 判例部門の配線用に `data-hyakusen="{科目}百選{巻}-{番号}"` が乗っている（正典 spec §11-1-ter）。
      **これを認識するのは `hyakusen-wire-ariadne.py` だけ**なので、case-card 開始タグを作り直す recanon／
      PDF 再生成は属性を脱落させうる。移行後は必ず
      **`python scripts/hyakusen-wire-ariadne.py --subject {科} --series {ariadne|tx} --apply`（冪等）で再配線**する
      （recanon 自体で属性を carry-over できるならそれでも可。生成プロンプトは §11-1-ter で最初から付与を指示済）。

## 4. アンチパターン早見

| やりがち（NG） | 正しい対処 |
|---|---|
| 「この既存ファイルを新版に直して」と LLM に手編集させる | 決定論的 recanon スクリプトで土台だけ載せ替え（本文不変） |
| 検証 ALL PASS だから完成とみなす | presence-PASS は見た目・機能を保証しない。整合ガード＋gold 目視で裏取り |
| 直らないので同じファイルに修正を重ねる | パッチにパッチ＝発散。canonical から土台ごと作り直す |
| 改定で spec/HTML だけ更新し検証は据え置き | 改定と同時に整合ガードを足し push 前ゲートに常駐させる |
| gold を作らず一括移行を開始 | 先に正実装を数問確保し、移植元・判別基準にする |
| recanon／再生成で case-card の `data-hyakusen` を落とす | 移行後に `hyakusen-wire-ariadne.py --apply` を再実行（冪等）して再配線 |
