# TX v12.2.1 inline 表示LOCK 正典（2026-07-01）

この文書は、TX355-359 の実地修正で合意した `_lex` 表示・操作・解説構造を、今後の既存ファイル更新と新規 TX 生成に必ず継承させるための正典である。

## 適用範囲

- 対象：`outputs/ux/000_TX/**/**/*_lex.html` のうち、`.tx-inline-card` を持つ v12 inline 型。
- 起点：`canonical/GENESIS-CORE.html`。
- 検証：`scripts/validate-tx-core.py` G45 と `scripts/check-tx-lex-engine.py`。
- 関連正典：`spec/tx-v12.1.0-inline-core.md`、`docs/tx-lex-two-track.md`、`docs/canonical-lineage.md`。

## 画面導線

- 問題文と ○× は一体化する。各記述は `.tx-inline-card[data-stmt]` に置き、○/× ボタンは `.tx-inline-actions` としてカード右端のバランスよい位置に寄せる。
- 裏の `.answer-ox-grid` は Lexia/SM2・answer-key・検証用のデータ源として DOM に残すが、画面の主導線に戻さない。
- `解法ナビ` は問題文の直下、インラインカード群の前に置く。空の飾り枠にせず、検討順・現在の設問・次の操作を読める状態にする。
- 解法ナビのヒントはネタバレ禁止。正誤・結論・正解番号を出さず、「どの客体を見るか」「条文上どの要件に当てるか」「判例のどの射程に注意するか」のような注目点にする。
- 全記述を選んだ後の `解答を表示` ボタンを欠落させない。`解説だけ閲覧` は answered を付けず、SM2 記録を発生させない。
- 旧 `FINAL ANSWER` 表は inline 画面に表示しない。`data-answer-key` などの抽出用 DOM としては残してよい。

## 条文・判例ボックス

- 各肢の `ANSWER` 箱の直後に、必ず `.tx-mini-law` を置く。文言・趣旨・射程だけで根拠を代用しない。
- 条文カードはブルー系、判例カードは `.is-case` のピンク系。上段チップはすべて重厚な楕円ピルにする。長方形・薄い枠だけのラベルへ戻さない。
- 上段チップに `題名：`、`テーマ：` という説明語を表示しない。
- 題名チップは下部 `basis-card-header` の参考条文・判例タイトルから取る。条文は罪名・制度名、判例は判例名や括弧内タイトルを使う。
- テーマチップは法理名にする。単なるカテゴリ名ではなく、例として `現住性錯誤`、`公共危険の非限定説`、`建造物の一部性` のように判断規範を示す。
- 本文冒頭の `本条` / `本文` / `①` / `②` / `判旨核心` / `判例` は `.tx-mini-law-para` で小ラベル化する。素文字にしない。
- 条文の項番号ラベル `①` / `②` は、2項以上ある条文でのみ使う。単一項または条文全体を示す場合は `本条` / `本文` を使う。
- `.tx-mini-law-para` を使う行は、本文を `.tx-mini-law-body` で包み、ラベル列＋本文列の2カラムにする。判例ボックスも同じ構造にする。
- 字下げの定義：段落本文の先頭文字だけを1字分空ける。折返し行は本文列にそろえ、ラベル下へぶら下げない。余分な予約スペースを作らない。
- 判例本文の先頭に `:` など不要な記号を入れない。

## 5点フロー・コツ・記憶フック

- 文言・趣旨・射程・切断点・転用の各ブロックは、ラベル列＋本文列の2カラムにする。
- 2カラム化された段落でも、本文先頭だけ1字分字下げする。折返し行は本文列にそろえる。
- `コツ` や `記憶フック` はラベルと本文を分け、ラベルだけが左にぶら下がる表示にしない。
- `コツ` は結論を言わない。問題文のどこに注目して考えるかを示す。

## 解説品質レビュー

- 解説は TX の学習価値そのものであり、機械的な整形・一括置換・テンプレ埋めだけで確定してはならない。
- 新規生成・既存更新・一括伝播のいずれでも、公開・push 前に、作業時点で利用可能な最上位モデルを使い、推論/エフォート設定は最大にして内容レビューを行う。
- 機械チェックは構造・表示・欠落・ネタバレ・旧UI残存の補助検査に限る。`validate-tx-core.py` や `check-tx-lex-engine.py` が PASS しても、解説内容の正確性が保証されたとは扱わない。
- レビュー対象は少なくとも、`ANSWER`、`.tx-mini-law`、`.tx-article-flow`、`コツ`、`記憶フック`、`詳説`、`物語解説`、SM2 用 `.ox-pool-explain` とする。
- レビューでは、問題文の事実、条文番号・項、判例名・判旨、結論、文言/趣旨/射程/切断点/転用の対応、正誤理由、非ネタバレ性を個別に照合する。
- 薄い一般論、結論の言い換えだけのコツ、条文・判例根拠の欠落、事案と条文の客体取り違え、軽い罪/重い罪の錯誤、現住/非現住などの法的分類ミスは、表示が整っていても不合格とする。
- 内容レビューが未了のファイルは、v12.2.1 完了・配信可・push可とは呼ばない。

## 物語解説

- 物語解説は初回理解用の読み物であり、通常の周回主導線ではない。
- `解答を表示` または `解説だけ閲覧` の前に物語解説を露出させない。
- reveal / browse 後は、インライン解説側にも物語解説が出るようにする。
- 物語解説の各ブロックは、ラベルと題名を持つカードにする。色はその問題の配色パターンに従う。
- `data-fa-label` ラベルは通常フロー内で左寄せにする。`position:absolute` や `left:` で浮かせない。本文と重ねない。
- 強調語は太くしすぎない。`.fa-narrative b` は `font-weight:560` 以下を維持する。

## 詳説・BASIS

- `詳説を開く` は空展開禁止。`details.tx-inline-detail` には必ず `.tx-detail-panel[data-partb-source]` を置く。
- 詳説最下部の条文・判例本文は、`BASIS` タブ付きの `sub-card basis-link` ボックス内に収める。枠・背景・タブを CSS リセットで消さない。

## 確認サーバー

- ローカル確認は `outputs/ux` をルートにして HTTP サーバーで見る。ファイル直開きでは相対パス・ブラウザ挙動が本番とずれるため、確認 URL を使う。
- 例：`python -m http.server 8788 --bind 127.0.0.1` を `outputs/ux` で起動し、`http://127.0.0.1:8788/000_TX/001_刑法/刑TX355_lex.html` を開く。
- この URL はこの PC 内だけで見える確認用 URL であり、公開 URL ではない。
