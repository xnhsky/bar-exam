# TX v12.2.1 inline 表示LOCK 正典（2026-07-01）

この文書は、TX355-360 の実地修正で合意した `_lex` 表示・操作・解説構造を、今後の既存ファイル更新と新規 TX 生成に必ず継承させるための正典である。

## 適用範囲

- 対象：`outputs/ux/000_TX/**/**/*_lex.html` のうち、`.tx-inline-card` を持つ v12 inline 型。
- 起点：`canonical/GENESIS-CORE.html`。
- 検証：`scripts/validate-tx-core.py` G45 と `scripts/check-tx-lex-engine.py`。
- 関連正典：`spec/tx-v12.1.0-inline-core.md`、`docs/tx-lex-two-track.md`、`docs/canonical-lineage.md`。

## 最終合意チェックリスト

更新・新規生成では、少なくとも次の状態を満たすこと。

- 問題文と○×ボタンは `.tx-inline-card` 内で一体化する。○/×ボタンは右端に寄せ、本文を不自然に圧迫しない。
- カード内の回答比較バッジは、○×ボタンの下に `あなた ×` / `正解 ○` の2行で表示する。`自分` 表記にはしない。バッジは薄い枠だけにせず、重厚な枠・影・背景を持たせる。
- 回答操作パネル内に大きな `正解/不正解`、`あなたの答え`、`正解`、`相違` のサマリー箱を出さない。
- トーストは使わない。回答後フィードバックは、カード内バッジと解説冒頭の比較用正誤表に集約する。
- 解説冒頭の比較用正誤表は、`あなた` 列を持たせる。ただし詳細解説表ではなく、各肢1文の「登場した論点のコア（転用可能な法理）」だけに圧縮する。
- 詳しい `文言・趣旨・射程・切断点・転用` は、表に詰め込まず、各設問の解説本文側に残す。
- 解法ナビは問題文直下、インラインカード前に置く。ヒントは結論・正誤・正解番号を漏らさず、「どこを見るか」を示す。
- 条文・判例ボックスは、見出しチップ、項ラベル、本文2カラム、本文先頭1字下げを維持する。判例も同じ扱いにする。
- 物語解説は、`解答を表示` または `解説だけ閲覧` の前に露出させない。表示後はラベル＋題名カードで、本文と被らない通常フローに置く。
- 文章本文の段落先頭は1字分空ける。ラベルやバッジに字下げを掛けず、本文列全体を余白で押し込まない。
- 詳説最下部の条文・判例本文は `BASIS` ボックスに収める。
- 凡例の `論` は「論点」ではない。論文試験と短答試験で重なる重要箇所に引くロンマークであり、凡例文言は `論文と重複` に固定する。
- 解説内容は機械的に確定しない。最新法令・判例・学説と、事案対応・条文番号・項・判例射程を最高エフォートで確認する。

## 画面導線

- 問題文と ○× は一体化する。各記述は `.tx-inline-card[data-stmt]` に置き、○/× ボタンは `.tx-inline-actions` としてカード右端のバランスよい位置に寄せる。
- 裏の `.answer-ox-grid` は Lexia/SM2・answer-key・検証用のデータ源として DOM に残すが、画面の主導線に戻さない。
- `解法ナビ` は問題文の直下、インラインカード群の前に置く。空の飾り枠にせず、検討順・現在の設問・次の操作を読める状態にする。
- 解法ナビのヒントはネタバレ禁止。正誤・結論・正解番号を出さず、「どの客体を見るか」「条文上どの要件に当てるか」「判例のどの射程に注意するか」のような注目点にする。
- 解法ナビの各 `STEP` / `STMT` には、`tip` だけでなく解答前表示用の `hint` を必ず置く。`hint` 欠落時のフォールバックに頼らず、各記述固有の分岐点を1つ以上入れる。
- 解法ナビ内に `.sn-answer-choices` を出し、現在STEPの○/×をその場で選べるようにする。下の一問一答へ移動しないと回答できない導線は禁止。
- `.sn-nav-ox[data-stmt][data-value]` は裏の `.answer-ox-grid .ox-row[data-stmt] .ox-btn[data-value]` をクリックして同期する。ナビ上の回答、下の一問一答、採点ボタンの有効化がズレてはならない。
- 全記述を選んだ後の `解答を表示` ボタンを欠落させない。`解説だけ閲覧` は answered を付けず、SM2 記録を発生させない。
- 旧 `FINAL ANSWER` 表は inline 画面に表示しない。`data-answer-key` などの抽出用 DOM としては残してよい。

## 条文・判例ボックス

- 各肢の `ANSWER` 箱の直後に、必ず `.tx-mini-law` を置く。文言・趣旨・射程だけで根拠を代用しない。
- 条文カードはブルー系、判例カードは `.is-case` のピンク系。上段チップはすべて重厚な楕円ピルにする。長方形・薄い枠だけのラベルへ戻さない。
- 上段チップに `題名：`、`テーマ：` という説明語を表示しない。
- 題名チップは下部 `basis-card-header` の参考条文・判例タイトルから取る。条文は罪名・制度名、判例は判例名や括弧内タイトルを使う。
- テーマチップは法理名にする。単なるカテゴリ名ではなく、例として `現住性錯誤`、`公共危険の非限定説`、`建造物の一部性` のように判断規範を示す。
- 条番号チップ（`.tx-mini-law-article`）は **ASCII 短縮形**にする（例 `112条` / `109条1項・2項`、判例は `最決平元.7.14`）。**漢数字（`第百十二条`）にしてはならない**。題名・法理テーマの導出はエンジンが ASCII 条番号の正規表現で行うため、漢数字だと導出が全滅し、本文を16字で切ったゴミチップ（例 `第百八条及び第百九条第一項の罪の`）が出る。本文（条文原文 `.tx-mini-law-body` / `.tx-mini-law-text`）は正式な漢数字のままでよい。`validate-tx-core.py` G45 が漢数字チップを検出する。
- 本文冒頭の `本条` / `本文` / `①` / `②` / `判旨核心` / `判例` は `.tx-mini-law-para` で小ラベル化する。素文字にしない。
- 条文の項番号ラベル `①` / `②` は、2項以上ある条文でのみ使う。単一項または条文全体を示す場合は `本条` / `本文` を使う。
- `.tx-mini-law-para` を使う行は、本文を `.tx-mini-law-body` で包み、ラベル列＋本文列の2カラムにする。判例ボックスも同じ構造にする。
- 字下げの定義：段落本文の先頭文字だけを1字分空ける。折返し行は本文列にそろえ、ラベル下へぶら下げない。余分な予約スペースを作らない。
- 判例本文の先頭に `:` など不要な記号を入れない。

## 論点処理マトリクス・5点フロー・コツ・記憶フック

- 論点処理の「整理図」は、各肢の `.tx-logic-matrix`（4セル処理順＋判断式）か、問題単位の `.tx-sysmap`（体系マップ）のどちらかで置く（validate G45 はどちらか一方があればよい）。5記述が同一の体系（限定列挙・所有区分など）を突く問題は、記述ごとに似たマトリクスを5枚並べるより、**体系マップ1枚に畳んで各記述を枝に対応づける**方がよい（情報整理・周回速度・転移の点で優位）。
- `.tx-logic-matrix` を採る肢では全肢に入れ、条文判例の直下・5点フローの直上に置く。ある肢だけ欠ける・条文判例より上・5点フローより下の配置崩れは禁止。`.tx-sysmap` を採る問題ではマトリクスを置かなくてよい。
- **整理図は焼き直し禁止（最重要）**：`.tx-logic-matrix` の各セル `.tx-matrix-body` を `.tx-flow-body`（5点フロー本文）の逐語コピーにしない（同一24字以上の連続一致で G45 ERROR）。判断式 `.tx-matrix-verdict` と記憶フック `.tx-op-body` を同一文にしない。マトリクスは「見る軸 → 確認事実 → 切る場所 → 結論到達」の上位地図、判断式は結論到達の決定式、記憶フックは1秒で思い出す標語に**役割分離**する。gold 見本＝刑TX366（逐語重複 1/20 以下）。
- `.tx-sysmap`（体系マップ）は記述リストの一番上（解説の先頭・記述1の直前）に置き、**解く間は非表示**にする。表示は `#part-a:has(.tx-inline-explain:not([hidden])) .tx-sysmap`（＋`[data-browse-open="true"]` フォールバック）の CSS フックで、「解答を表示／解説だけ閲覧」時に現す（エンジンJSは無改変＝接ぎ木禁止）。各記述タグ（`a.tx-sysmap-probe` → `#stmt-N`）と各解説末尾の戻りリンク（`.tx-sysmap-back` → `#tx-sysmap`）でハブ往復させる。見出しラベル（`.tx-sysmap-section-label`＋番号バッジ）・ノードのヘッダー帯・付属犯リストの交互背景（ゼブラ）・テーマカラー（`--accent`/`--mid`）で重厚に作る。gold 見本＝刑TX360。
- 文言・趣旨・射程・切断点・転用の各ブロックは、ラベル列＋本文列の2カラムにする。
- 2カラム化された段落でも、本文先頭だけ1字分字下げする。折返し行は本文列にそろえる。
- `コツ` や `記憶フック` はラベルと本文を分け、ラベルだけが左にぶら下がる表示にしない。
- `コツ` は結論を言わない。問題文のどこに注目して考えるかを示す。
- `記憶フック` は解説要約ではない。論点のコア・テーゼを15〜45字程度の一言にし、55字を超える判例名・要件・結論の連結文へ戻さない。

## 解説品質レビュー

- 解説は TX の学習価値そのものであり、機械的な整形・一括置換・テンプレ埋めだけで確定してはならない。
- 新規生成・既存更新・一括伝播のいずれでも、公開・push 前に、作業時点で利用可能な最上位モデルを使い、推論/エフォート設定は最大にして内容レビューを行う。
- 機械チェックは構造・表示・欠落・ネタバレ・旧UI残存の補助検査に限る。`validate-tx-core.py` や `check-tx-lex-engine.py` が PASS しても、解説内容の正確性が保証されたとは扱わない。
- レビュー対象は少なくとも、`ANSWER`、`.tx-mini-law`、`.tx-logic-matrix`、`.tx-article-flow`、`コツ`、`記憶フック`、`詳説`、`物語解説`、SM2 用 `.ox-pool-explain` とする。
- レビューでは、問題文の事実、条文番号・項、判例名・判旨、結論、文言/趣旨/射程/切断点/転用の対応、正誤理由、非ネタバレ性を個別に照合する。
- 薄い一般論、結論の言い換えだけのコツ、条文・判例根拠の欠落、事案と条文の客体取り違え、軽い罪/重い罪の錯誤、現住/非現住などの法的分類ミスは、表示が整っていても不合格とする。
- 内容レビューが未了のファイルは、v12.2.1 完了・配信可・push可とは呼ばない。

## 最新法令・判例・学説レビュー

- TX は司法試験対策教材なので、生成・更新時点の最新法令、最新判例、主要学説の現状を必ず確認する。
- 法令は e-Gov 法令検索（`https://laws.e-gov.go.jp/`）を一次確認先とし、現行条文、未施行改正、施行日、改正前後の文言を確認する。
- 司法試験・予備試験の出題基準日は、法務省の当該年度の司法試験/予備試験実施情報、Q&A、試験用法文登載法令を確認する。教材の基準日は「生成日現在」ではなく、対象試験年度の法令基準に合わせる。
- 判例は裁判所の裁判例検索（`https://www.courts.go.jp/hanrei/search1/index.html`）と最近の裁判例（`https://www.courts.go.jp/hanrei/index.html`）を確認する。裁判所サイトは全判決を網羅しないため、最高裁判例集、判例百選、重要判例解説、主要判例データベース等で補完する。
- 学説は公式の単一ソースがないため、最新版の基本書、判例百選、重要判例解説、法学教室・法学セミナー等の最新解説、受験年度向け改正情報を確認する。
- 旧法・旧判例・旧説に基づく過去問を扱う場合でも、現在法での処理を必ず確認し、必要なら `tx-current-law-note` コラムを入れる。
- `tx-current-law-note` には、確認日、参照ソース、旧法/旧判例/旧説、現在法/最新判例/現在の有力説、立法経緯・改正経緯、改正趣旨、試験対策上の扱いを短く書く。
- `tx-current-law-note` の中身は `.tx-law-note-row` で、左に `.tx-law-note-label`、右に `.tx-law-note-body` を置く2カラムにする。ラベル列は `grid-template-columns:max-content minmax(0,1fr)` で実幅に寄せ、本文との間を空けすぎない。ラベルは字下げせず、本文側だけ各段落の先頭を1字下げする。
- 新判例や改正で結論・理由付け・用語が変わる場合は、本文を最新化し、旧処理は「旧法下の過去問理解」として補足に退避する。新旧どちらで答えるべきか曖昧な状態で配信しない。
- 最新確認が未了のファイルは、内容レビュー未了と同じく、v12.2.1 完了・配信可・push可とは扱わない。

## 物語解説

- 物語解説は初回理解用の読み物であり、通常の周回主導線ではない。
- `解答を表示` または `解説だけ閲覧` の前に物語解説を露出させない。
- reveal / browse 後は、インライン解説側にも物語解説が出るようにする。
- 物語解説の各ブロックは、ラベルと題名を持つカードにする。色はその問題の配色パターンに従う。
- `data-fa-label` ラベルは通常フロー内で左寄せにする。`position:absolute` や `left:` で浮かせない。本文と重ねない。
- ラベル付き段落の本文は必ず `.fa-narrative-body` に包み、本文側だけ `text-indent:1em` で1字下げにする。ラベル・見出し・バッジは字下げ対象ではない。
- 強調語は太くしすぎない。`.fa-narrative b` は `font-weight:560` 以下を維持する。

## 記憶フック・回答フィードバック

- 記憶フック `.tx-onepoint .tx-op-body` は本文列全体を `padding-left` で押し込まない。本文先頭だけ `text-indent:1em` で1字下げにする。
- `解答を表示` 後、回答操作パネル内に大きな「あなたの答え／正解／相違」サマリー箱を出さない。回答比較はカード内判定欄と解説冒頭の正誤表に集約する。
- 問題カード内の判定欄は、○×ボタンの下に2行で出す。本文列を圧迫しないため、`あなた ×` / `正解 ○` のように結論記号だけを短く表示し、バッジは重厚な枠・影・背景で視認性を確保する。
- 解答後・解説だけ閲覧後は、解説冒頭に `.statement-verdict-table` を複製した正誤表を常時表示し、`あなた` 列を追加してユーザー回答と正解を比較できるようにする。
- 解説冒頭に表示する正誤表は詳細解説ではない。`文言・趣旨・射程・切断点・転用` の全行をそのまま出さず、各肢1文の「登場した論点のコア（転用可能な法理）」に圧縮する。詳しい内容は各設問の解説本文に残す。
- トーストは廃止する。回答後フィードバックは、解説冒頭の正誤表とカード内2行判定に集約する。
- ユーザー回答が正解の行は緑背景、誤答の行はピンク背景、未回答は淡色で表示する。

## 詳説・BASIS

- `詳説を開く` は空展開禁止。`details.tx-inline-detail` には必ず `.tx-detail-panel[data-partb-source]` を置く。
- 詳説最下部の条文・判例本文は、`BASIS` タブ付きの `sub-card basis-link` ボックス内に収める。枠・背景・タブを CSS リセットで消さない。

## 確認サーバー

- ローカル確認は `outputs/ux` をルートにして HTTP サーバーで見る。ファイル直開きでは相対パス・ブラウザ挙動が本番とずれるため、確認 URL を使う。
- 例：`python -m http.server 8788 --bind 127.0.0.1` を `outputs/ux` で起動し、`http://127.0.0.1:8788/000_TX/001_刑法/刑TX355_lex.html` を開く。
- この URL はこの PC 内だけで見える確認用 URL であり、公開 URL ではない。

   （記述番号）に `text-align:center; vertical-align:middle` を与える。JS 正誤表
