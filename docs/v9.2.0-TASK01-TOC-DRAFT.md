## 5. 設計判断の根拠記録

### 5-1. なぜ §17-ter で C-4 deep-dive を独立 spec section 化したか

現行 spec で §17 = PART A、§17-bis = PART C content wrapper という命名規則がある。C-4 は PART C 内の section なので、§17-ter として並べることで「PART 構造系 canonical lock の系譜」に位置づける。§22 系（SVG/final-answer の系譜）とは性質が異なる（§17-ter は HTML 構造、§22-* は SVG 構造）。

### 5-2. なぜ §34-decies を意図的に欠番にしたか

handoff §6-2 で「既存 14 ファイルは v9.1.0 / v8.11.7 のまま維持」「305 以降の新規生成は v9.2.0」と確定済み。v9.1.0 → v9.2.0 のインプレース minor 更新パスを提供しないことを spec で明示する方が、将来の混乱を防ぐ。

### 5-3. なぜ §32 を「拡張」扱いにしたか（新規セクション化しなかったか）

§32 は「3 パターン色変換ルール（P1/P2/P3）」で、派生色変数は P1/P2/P3 各々で値を持つので、§32 の内部拡張として整理するのが構造的に正しい。Annex A 側の実装と §32 側の運用ルールが対応する。

### 5-4. なぜ §36-bis を新設するか

§36 = v8.11.7 設計総括。これを書き換えると過去の意思決定記録が失われる。v9.2.0 は別の §36-bis として新設し、依頼 7 項目への対応マトリクスを残す。
