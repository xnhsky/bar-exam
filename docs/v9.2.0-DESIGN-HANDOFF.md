## 4. 現状 spec の問題点分析（前チャットで実施済み）

各改訂依頼項目について、現状の `spec/tx-v9.1.0-mindmap-core.md` で何が足りていないかの分析。

### 4-1. KEY フレーズ問題（依頼 #4）

- spec § 0-quad-3 STEP IQ-5 で「核心命題を 2〜3 行で凝集」と書いてあるが、**「解答選択肢の選び方への言及禁止」のような明示的ネガティブガードがない**
- 結果として AI が「肢4を選ぶ際は…」「正解は記号〇である理由は…」のような**メタ説明**を書きがち
- **解消方針**: § 0-quad-2 ブラックリストに「メタ説明文言カテゴリ」を新設。検出語句例：
  - 「肢を選ぶ／選択する／選び出す」
  - 「正解は～／本問の正解は」
  - 「記号［ア／イ／ウ］を選ぶ／選択する／選定する」
  - 「解答の手順／プロセス」
- 加えて § 0-quad-3 に STEP IQ-8（メタ説明検閲）を追加：執筆中に上記語句が浮かんだら停止して核心命題に書き直す

### 4-2. マインドマップ問題（依頼 #5）

依頼 PDF は明確に**別構造の 2 図**を要求。現行 § 22-quad は中途半端なハイブリッド（中心 → 4 体系層 → 各記述+条文+判例）でどちらも満たさない。

**解消方針: § 22-quad を 2 セクションに分割**

| 新セクション | 構造 | 配置 |
|---|---|---|
| § 22-tree（仮称） | **ツリー型体系図**（縦/横階層、財産犯 → 罪種 → 構成要件） | basis section 直後 |
| § 22-radial（仮称） | **放射状マインドマップ**（中心法理 → 構成要件①〜④／保護法益／法定刑／本問論点／特別法均衡） | § 22-tree の直後 |

両方を独立 `<section>` として PART C-1 の前に並べる。`section id="mindmap"` は後方互換のため `section id="mindmap-tree"` `section id="mindmap-radial"` に分割するか、`mindmap` 内に 2 figure-wrap を入れ子にするかは設計時判断。

**SVG 設計上の注意**:
- ツリー型：viewBox は問題ごとの階層深度に応じて可変（例：1100×600 標準、深い体系なら 1100×800）
- 放射状：viewBox 1100×900（現行 § 22-quad のまま継承）
- 両方とも CSS 変数経由でパレット連動（v8.11.0 以来の規律）
- 「本問の論点」枠を必ず含める（依頼 PDF 1 枚目で明示されている要素）

### 4-3. 総合フローチャート問題（依頼 #6）

現行 § C-5 は **直線型** stepbox スタック（条件分岐なし）。依頼 PDF 3 枚目の構造：

- **decision diamond**（菱形ノード：判定基準の質問）
- **分岐**（左：成立しない／右：次のステップへ、または逆）
- **肢マッピング**（「肢2：河原放置=無主物」等を分岐線上にラベル配置）
- **終端ノード**（緑：成立／赤：不成立、左右に分かれる）
- **START ノード**（角丸長方形、紺色）

**解消方針: § C-5 を全面再設計**

新スキーマ（§ 22-? として番号採番、設計時に決定）：

```html
<svg viewBox="0 0 900 800">
  <defs>
    <linearGradient id="diamondGrad">...</linearGradient>
    <linearGradient id="endSuccessGrad">緑系</linearGradient>
    <linearGradient id="endFailGrad">赤系</linearGradient>
    <marker id="branchArr">...</marker>
  </defs>
  <!-- START ノード -->
  <g class="flow-start">...</g>
  <!-- decision diamond × N -->
  <g class="flow-decision">
    <polygon points="..." fill="url(#diamondGrad)"/>
    <text>① 客体は他人の物か？</text>
  </g>
  <!-- 分岐線（左右、肢ラベル付き） -->
  <g class="flow-branch">
    <path d="..." stroke="..."/>
    <text class="branch-label">肢2：河原放置=無主物</text>
  </g>
  <!-- 終端ノード × 2（成立/不成立） -->
  <g class="flow-end-success">...</g>
  <g class="flow-end-fail">...</g>
</svg>
```

CSS は § Annex A に追加（`.flow-decision`, `.flow-branch`, `.flow-end-success`, `.flow-end-fail`, `.branch-label`）。

**配置上の選択肢**:
- A. § C-5 を分岐型に全面置換（後方互換破壊）
- B. § C-5 直線型を残し、新規 § C-5-bis として分岐型を追加
- C. § C-5 内で SVG 種別を data 属性で切替

xnh の回答が「A 全面再設計」なので **A** が確定路線。ただし v9.2.0 ファイルは新規生成分のみに適用なので、v9.1.0 までの既存 § C-5 直線型は変更不要。

### 4-4. ベースカラー問題（依頼 #2）

- 現状 § Annex A の 27 色変数は P1/P2/P3 で全置換する設計
- 各パターン内では限定された色しか使われない構造
- 多様性を出すには：
  - 派生色変数を増やす（accent-light, accent-darker, mid-warm, mid-cool 等）
  - もしくは各セクションで同系統 mixin を許容

**JX v3.2 付録 A・B の TX 移植が解決策**:
- JX v3.2 付録 A：「同系統調和」「寒色×暖色対比」「紙質風」「黒板風」の 4 戦略
- JX v3.2 付録 B：書体組合せ判断（display/body/keyword の差替条件）
- これらが TX 側にまだ存在しない

**解消方針: v9.2.0 で TX § Annex A に「AI 自由設計運用条項」を新設**
- P1/P2/P3 の 3 パターンは骨格として維持
- 各パターン内で「同系色派生」「セクション別 mixin」を spec で許可
- ただし v8.11.0 規律 S60「override 形式は単一 :root{} のみ」は維持 → 派生色を最初から `:root` 内に増やす形で対応
- 派生色変数の命名規則を spec で標準化（例：`--accent-light`, `--accent-darker`, `--mid-warm`, `--mid-cool` 等 8〜12 個）
- 各セクション/カードで派生色をどう使うかの**運用ガイド表**を追加

### 4-5. 学説問題深堀り問題（依頼 #7）

現状 § C-4「学説対立」は `cmp-table-wrap`（学説／結論／論拠の 3 列表）のみ。依頼の深堀りには以下が不足：
- 判例採用の理由（なぜその学説か）
- 対立構造の詳細（学説間の論点軸：要件論／効果論／手続論）
- 条文の論点深堀り（条文文言の解釈論争）

**解消方針: § C-4 スキーマ拡張**

```
§ C-4 学説対立（拡張版）
├── 対立表（既存 cmp-table-wrap）
├── 各説の詳説 sub-card（新規）
│   ├── 通説/判例 sub-card
│   │   ├── 結論
│   │   ├── 論拠（条文解釈・体系的理由）
│   │   ├── 判例が採用する理由（新規必須項目）
│   │   └── 批判への応答
│   └── 少数説 sub-card
│       ├── 結論
│       ├── 論拠
│       ├── 判例が採用しない理由（新規必須項目）
│       └── 実務上の問題点
├── 対立軸の図解（任意 SVG）
└── 条文文言解釈 blockquote（新規・条文の論点に焦点）
```

加えて、学説問題（語群から記号を選ぶ型）が出題された場合は、§ 0-bis STEP 4「問題形式判定」で `data-question-type="theory-selection"` 等のフラグを立て、prompt 側で「メタ説明禁止・条文判例深堀り必須」を強制する。

### 4-6. 教授解説密度問題（依頼 #3）

現状 spec の `.sub-card.professor` 4 prof-heading（ポイント／考え方の道筋／イメージで掴む／あてはめ）構造は維持。問題は**密度**と**初学者対応**。

**解消方針: § 0-quad-3 STEP IQ-5 を強化**
- 各 prof-heading 配下の最小文字数を明示（例：「考え方の道筋」は条文構造から判例規範まで最低 400 字）
- 「イメージで掴む」セクションに**必須要素**を追加：
  - 具体場面の描写（日常生活レベルの例示）
  - 規範への接続（イメージ → 法律論への橋渡し）
  - 反対結論との対比（なぜ別の結論にならないか）
- 「ポイント」は条文構造の図解的整理を含むこと（箇条書きまたは小型 SVG）
- 「あてはめ」は本問事案を判例規範に1ステップずつ照らす形式（要件 → 該当性 → 結論の3段論法を可視化）

### 4-7. 全体完璧化（依頼 #1）

これは抽象方針なので、上記 4-1〜4-6 の総和として実現される。spec 単独の項目化は不要だが、v9.2.0 spec の冒頭ステートメントに「**single-document-self-sufficient-deep** = HTML 1 枚で基本書を不要にする情報密度」を新規 feature-tag として追加。

## 7. 設計タスクリスト（次セッションで実施）

優先順位順。各タスクの完了基準と STOP-for-review ポイントを明示。

### Task 1: v9.2.0 spec 全体章立て確定
- § 22-tree / § 22-radial / § 22-flowchart-v2 の章番号採番

- **完了基準**: spec ファイルの目次レベルが確定
- **STOP**: 章番号採番完了時に xnh レビュー

### Task 2: § Annex A 派生色変数の設計
- 派生色変数の命名規則（例：`--accent-light`, `--accent-darker` 等）
- P1/P2/P3 各々で派生色を具体的にどの色値にするか
- AI 自由設計運用条項の文面起草
- セクション/カード別の派生色適用ガイド表
- **完了基準**: P1 についての派生色値が全部決まり、運用ガイド表ドラフトができる
- **STOP**: P1 派生色決定時に xnh レビュー（色覚的に xnh の好みに合うか）

### Task 3: § 22-tree（ツリー型体系図）SVG 仕様
- 階層レイアウト（縦/横）の選択

- viewBox 可変ルール（階層深度に応じて）

- 検証規律 S85 文案
- **完了基準**: SVG スケルトンと CSS class 一覧が確定
- **STOP**: スケルトン完成時に xnh レビュー（依頼 PDF 1 枚目と比較）

### Task 4: § 22-radial（放射状マインドマップ）SVG 仕様
- 現行 § 22-quad の骨格を継承しつつ詳細化
- 中心ノード → 主要枝（保護法益／構成要件①〜④／法定刑／本問論点／特別法均衡）の配置

## 9. 既知の運用上の注意（memory より）

- **PowerShell**: pwsh 7.6.2 使用、.ps1 ファイルは BOM 付き UTF-8 必須
- **Windows cp932 回避**: Python スクリプトは `sys.stdout.reconfigure(encoding='utf-8')` 必須
- **Japanese 名 zip 回避**: Linux 上で zip 作成時は日本語ディレクトリ名禁止（`刑TX/` → `#U5211TX/` 化）
- **validate_content.py の auto-quarantine 罠**: 全ファイル一括検証で 刑TX300.html が `_quarantine/` に移動し CP gate 破壊。個別ファイルで実行
- **claude -p (headless) コスト**: 1 問あたり API 換算 $5-7（Opus 4.7, 1M context, output ~93K tokens）。Max 20x なら定額 $200/月で追加課金なし、レートリミット制
- **night-batch-runner**: 5 問バッチ、平均 24.0 min/問。`-DryRun` `-MaxProblems` `-MaxConsecutiveFailures` パラメータあり、cost-summary.csv 累積記録
- **stored memory vs actual file state divergence**: 多 PC 環境で memory と実ファイルが乖離することあり。常に actual file state 優先
