## 2-bis. 短答反射の5点セット（TX-LEX 正典ルール）

TX-LEX は「正解根拠を読む教材」ではなく、Lexia/SM2 で回すための **誤肢切断・条文判例反射装置** とする。`.ox-stmt` と verdict「論点のコア」は、過去問の肢を次回以降の類似肢に転用できる形で圧縮する。

各肢のコアには、必要に応じて次の5点を入れる。全部を長く書くのではなく、**1肢1ストーリー／1肢1切断点**でまとめる。

| 要素 | 役割 | 書き方 |
|---|---|---|
| **文言** | 条文のどの語から出発するか | 「○条の『〜』は…」 |
| **趣旨** | なぜその読み方になるか | 制度趣旨・保護法益を一文で置く |
| **射程** | 判例・通説がどこまで言っているか | 事案差・要件差を短く示す |
| **切断点** | 誤肢ならどの語で落とすか | 「危ない語：常に／直ちに／当然に／一切」などを明示 |
| **転用** | 次の類似肢でどこを見るか | 「次は○○の有無を見る」と書く |

- 誤りの肢では、必ず「危ない語」または「過剰一般化」を特定する。
- 正しい肢では、切断点の代わりに「肯定条件」「成立条件」を明示する。
- 解法テクニック（組合せ消去・逆算・選択肢番号の処理）は、`.ox-stmt` に混ぜず、解法ナビに置く。
- 短答主力教材として、説明は長文化しない。移動中に1画面で想起できる密度を上限とする。
- 「判例によれば正しい／誤り」で終わらせない。必ず、条文文言・保護法益・判例射程のどれで正誤が決まるかを残す。

悪い例：

> 判例に照らすと誤り。

良い例：

> **文言**：○条は「〜」を要件にする。**趣旨**：保護法益は〜だから、単なる〜だけでは足りない。**切断点**：「当然に」が過剰。**転用**：次は「〜の認識／危険の現実化」の有無を見る。

## 2-ter. TX-LEX 標準構成（必須・条件付き・補助掲載）

TX-LEX の主目的は、短答で肢を見た瞬間に **条文・判例・危ない語で切れる** ようにすること。したがって、周回導線の中心は次の5項目に固定する。

| 区分 | 要素 | 位置づけ |
|---|---|---|
| **必須** | ox-grid | 肢ごとの正誤判断を先に強制する主エンジン |
| **必須** | 自己完結命題 | Lexia/SM2 で単独復習できる知識単位 |
| **必須** | 正誤 | その場で○×を確定し、誤答を潰す |
| **必須** | 文言・趣旨・射程・切断点・転用 | 誤肢切断と類似肢への転用を作る中核 |
| **必須** | 解法ナビ | 組合せ・消去・検討順を案内する。ただし知識カードには混ぜない |
| **条件付き** | 横断ミニ | 頻出混同だけを2〜3行で整理。長くしない |
| **条件付き** | 比較ミニ表 | Aなら○／Bなら×が一瞬で見える場合に限る |
| **条件付き** | 条文・判例カード | 文言・保護法益・判例射程が肢の正誤に直結する場合に置く |
| **補助掲載** | 大きいマインドマップ | 削除しないが、周回の主導線にはしない。直前確認・迷子防止用 |
| **補助掲載** | 長いストーリー解説 | 削除しないが、通常周回では読ませない。最後の確認用 |
| **補助掲載** | 深掘りコラム | 削除しないが、JX/ATHENA 側に寄せる。TX-LEX では補助に留める |
| **補助掲載** | 論文答案風の厚い説明 | 削除しないが、ARIADNE/RX の領域。TX-LEX では主教材化しない |

- TX-LEX は美しくても「読む教材」になったら負け。画面上の主役は常に ox-grid と反射コア。
- 横断・比較・条文判例カードは、短答の切断速度を上げるときだけ使う。
- マインドマップ・ストーリー・深掘り・論文答案風説明は、載せてよい。ただしページ後半・補助セクション・必要時参照の位置づけにする。
- 補助掲載要素を入れる場合も、title/code/fileName/footer/data-rx/data-athena-code に影響させない。

## 2-quater. TX360 型インライン周回正典（2026-06-29）

TX-LEX の通常周回は、問題文を読んだ後に下の ox-grid へ移動して解く形ではなく、**問題文各肢そのものに OX と解説を対応**させる。これにより「肢を見る → その場で○× → その場で条文・5点セット・記憶フックを確認」という一画面導線にする。

正典構造は `刑TX360_lex.html` を gold とし、`canonical/GENESIS-CORE.html` に `tx-inline-*` CSS/JS を吸収済み。

| 要素 | 位置づけ |
|---|---|
| `.tx-inline-card` | 問題文の各肢。右側に ○/× ボタンを置き、裏の `.answer-ox-grid` と同期する |
| `.tx-mini-law` | その肢に登場する条文本文。メイン条文はブルー系、補助条文は薄め、周辺条文はグレー |
| `.tx-article-flow` | 文言・趣旨・射程・切断点・転用を肢ごとに統合して読む本体 |
| `.tx-cycle-aids` | 記憶フック（紫系）と答案圧縮（ピンク系）。いずれも短く、ネタバレ目的ではなく記憶定着用 |
| `.tx-inline-detail` | 「詳説を開く」で対応する PART B 詳説をその場に展開するトグル |
| `.tx-inline-browse-btn` | 全5肢を選ばずに「解説だけ閲覧」。answered にせず、Lexia/SM2 の記録は発生させない |
| `.answer-area.inline-prototype-mode` | 裏の ox-grid。Lexia 記録・answer-key・検証の単一情報源として DOM に保持する |
| `.ox-pool-explain` | 裏の各 `.ox-row` 直下に置く SM2 専用 GIST/POINT。既存 Lexia 抽出器が最優先で読むため、`.tx-reflex-core`＋`.tx-cycle-aids` をここへミラーする |

運用ルール:

- インラインカードが周回の主導線。下の素の ox-grid は表示上は隠してよいが、Lexia 記録のため削除しない。
- 「詳説を開く」は対応する PART B をトグル表示する。したがって PART B は実質的には後置きの詳説アーカイブとなり、通常周回の本文ではない。
- `.final-answer` 冒頭の古い正解要約（fa-summary）は置かない。順序は **物語解説 → 論点コア表**。
- 共通根拠条文・判例セクションとアンカーリンクは残す。リンクジャンプはインライン詳説内でも機能させる。
- 物語解説は初回理解用の読み物であり、強調語は太くしすぎない。`.fa-narrative b` は `font-weight:560` 以下を正典値とし、モバイルで潰れる700系へ戻さない。

2026-07-01 追補（TX355-359 実地修正で確定した LOCK）:

- `.final-answer` は `data-answer-key` と検証のため DOM には残す。ただし `answer-area.inline-prototype-mode` 内では画面表示しない。古い `FINAL ANSWER` パネルを採点後に露出させない。
- 回答 UI はインラインカード側を主導線にする。全肢回答後の「解答を表示」導線を欠落させない。旧 ox-grid / 旧 reveal UI はデータ源として残すが、画面上の主役に戻さない。
- `tx-mini-law` は CSS だけでなく各肢のHTML本体に必ず置く。ANSWER の直後、5点フローの前に、該当条文本文・判例要旨を `tx-law-item` として表示する。
- `tx-inline-detail` の最下部に置く `sub-card basis-link` は、必ず `BASIS` タブ付きのボックスとして見せる。`.tx-detail-partb` のリセットで枠・背景・タブを消さない。条文・判例本文はこのボックスの中に収める。
- 詳説は空にしない。`details.tx-inline-detail` には必ず `div.tx-detail-panel.tx-detail-partb[data-partb-source="N"]` を置き、PART B 詳説をその場で展開できる状態にする。
- 条文本文の項バッジは、2項以上ある条文だけ表示する。条文本文横に `本文` / `①` / `②` ... のバッジを置き、単一項の条文には付けない。
- 記憶フックはラベルと本文を分離し、2カラムまたはラベル列＋本文列で字下げする。ラベルだけが左にぶら下がる表示にしない。
- 5点フロー・条文項バッジ・機能ラベルは正典の楕円ピルを使う。角丸長方形へ退行させない。
- 解法ナビは空の飾り枠にしない。検討順・判定コア・次の操作が読める状態を維持し、下の回答 UI と矛盾させない。

## 2-quinquies. 問題都合ラベルから論点コア・テーゼへの置換

記号問題、組合せ問題、見解A/B、事例Ⅰ/Ⅱ、空欄①〜⑤などは、**問題を成立させるための包装**であって、SM2 に残す知識ではない。TX-LEX では、これらをそのままカード化せず、必ず **論点コア・テーゼ** に置換する。

| 問題都合の表現 | SM2/論点コアでの扱い |
|---|---|
| `①b` / `(a)` / 組合せ番号 | 語句・制度・要件の実体名へ置換 |
| `A説` / `B説` / `甲説` | 学説名、または見解の定義内容を主語にする |
| `記述ア` / `本問` / `上記見解` | その肢の法命題として単独で読める文へ直す |
| 正解番号・消去順 | 解法ナビ・詳説に置き、SM2カード本文には入れない |

基準は「翌月、問題文を見ずにそのカードだけを開いても、何の論点を何で切るのかが分かること」。カードに残すのは、番号ではなく、文言・趣旨・射程・切断点・転用である。

## 2-sexies. Lexia/SM2 解説ペイロードの優先順位

Lexia の SM2 で表示・保存する解説は、長い読み物ではなく、復習時に再判定できる最小セットに寄せる。

| 優先 | SM2 に載せる情報 | 理由 |
|---|---|---|
| 1 | `.ox-stmt` | 単独で○×判定できるカード本文 |
| 2 | `.ox-pool-explain`（中身は `.tx-reflex-core` 5点セット） | 既存 Lexia が GIST/POINT として最優先抽出するカード裏面 |
| 3 | `.ox-pool-explain` 内の記憶フック/答案圧縮（元 `.tx-cycle-aids`） | 周回時の想起補助。短く、結論の丸暗記にしない |
| 4 | 対応する条文番号・メイン条文の本文 | 文言から切る問題で必須 |
| 5 | `.tx-inline-detail` / PART B 詳説 | 誤答時・理解不能時の展開情報。通常のSM2表面には載せすぎない |

`fa-narrative` は初回理解・全体像の救済テキストであり、SM2 の通常カード本文には載せない。SM2 は「読む場所」ではなく「再判定する場所」とする。

実装メモ：`scripts/tx-sm2-payload-backfill.py` が TX360 inline 型の `_lex` から `.tx-reflex-core` と `.tx-cycle-aids` を抽出し、各 `.ox-row` 直下の `.ox-pool-explain` へ冪等注入する。`validate-tx-core.py` G34 が `.ox-pool-explain` の欠落、`fa-narrative`/詳説混入、問題都合ラベル残留を ERROR にする。

## 3. 350方式（組合せ型 Lexia用）の構造

各空欄①〜⑤＝1 ox-row。3要素：
1. **設問** `.ox-q`（解く）
2. **過去問選択肢** `.ox-btn`（`data-value`=letter・表示のみ・記号OK）
3. **自己完結命題** `.ox-stmt`（`.ox-core-wrap` 内・reveal 後に展開・**プールに入るのはこれだけ**）

- `data-correct-value` は letter 連結（例 `bcfgj`）。**既存採点エンジン（handleOxBtn/handleRevealAnswerBtn・value非依存）がそのまま採点**。
- verdict 表は「空欄｜正解｜論点のコア」。`data-answer-key="1:b,2:c,3:f,4:g,5:j"`（letter）で G23 充足。
- **解法ナビ**＝回答ボタンを持たず、検討順（ORDER）＋💡コツ＋組合せ消去パネルでガイド。ユーザーは下の ox-grid で逐一回答→候補が絞られる→最後に reveal。
- 配色：正解＝**淡緑地＋濃緑字**（design system）。コツ・論点のコアは**ラベル見出し＋本文を次行で字下げ（1.5em ハンギング）**。

## 3-bis. ○×方式（通常型 Lexia用・001 基準）

- ox-grid は各記述 ○/×（既に「選択肢＝○×」で自己完結）。
- 解法ナビは○×型エンジン（記述を順に判定→「正しい/誤っているものはどれか」の候補から消去→残り番号→確定で採点）。
- mode は `data-correct-value` の ○/× 個数で自動判定（○1個=正しいものはどれか／×1個=誤っているものはどれか）。

6. **記号のみ6問（114/119/174/218/344/351）を最優先**で350化（現状プール価値ゼロ）。

- 参照スクリプト：`scripts/lex/build-ox-lex.py`（○×型＝タイプ④の一括ビルダ雛形。既存 ox-grid から `_lex`＋公式を生成、mode は ○/× 個数で自動判定）。**注：内部 `ROOT` がリモート用 `/home/user/bar-exam` 固定＝ローカル実行時はパス調整が必要**。組合せ型(350)の変換手順は本書 §3。
- ナビ文面（💡コツ・論点のコア命題）は問題ごとに執筆が必要（量が多い＝複数セッション／バッチ）。
