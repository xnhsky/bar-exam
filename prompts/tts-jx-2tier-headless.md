## Section 3: 二段構成の仕様（047 に準拠・論点数/被告人数で自動適応）

> 047 と design-spec の差分メモ：**二段構成では冒頭末尾の「超スパルタ喝入れ」rant は使わない**
> （047 は不採用）。カリスマコーチの砕けたタメ口・たとえ話・問いかけ（アクティブリコール）は維持し、
> 初級編 01 は穏やかなテーマ導入で始め、上級編 総まとめは想起クイズ＋やさしい励ましで締める。

### 初級編（理解トラック）= `{OUTPUT_DIR}\初級編\`
学習者が論点を理解するトラック。問題の論点数を N とすると **(3 + 2×N) 本**：

1. `01_問題文.txt` — 開き「問題文。」→ 今日のテーマ＋この一問で得られること2〜3点＋
   問題文を**HTML 原文どおり完全朗読**（`#case-overview` の `blockquote.problem`）→ 次への一言。
2. `02_事案分析.txt` — 開き「事案分析。」→ 登場人物・関係・時系列・トリガー事実を口頭で再構成
   （`#relationship-diagram`/`#timeline`/`#fact-analysis` 系）。ネタバレ（論点名・規範名）は伏せる。
3. `03_論点抽出.txt` — 開き「論点抽出。」→ 出る論点を問いかけで挙げさせ、番号付き＋配点感で整理
   （`#issue-extraction`）。
4. 以降、**各論点 K（1..N）について 2 本ずつ**：
   - `0{m}_規範{K}_{論点名}.txt` — 開き「規範{漢数字}、{論点名}。」→ 問題の所在→規範コア朗読
     （G頻出論証ブロックの規範を文語調で）→条文/趣旨/判例（年月日正確）→**たとえ話必須**→反対説一言→
     規範の再唱促し。（047 の規範ファイルの厚み＝3500〜4200字を目安）
   - `0{m}_あてはめ{K}_{論点名}.txt` — 開き「あてはめ{漢数字}、{論点名}。」→ 認識事実/発生事実を
     具体引用→事実→評価語→結論の3点セットを複数実演→反対方向の事実→結論明示。（2800〜3300字目安）
   - `{論点名}` は本問固有の短い日本語（例「事後強盗の身分」「承継的共同正犯」）。`{m}` は初級編内の通し
     2桁ゼロ埋め（04,05,06,07,…）。`{漢数字}` は一,二,三…。

### 上級編（答案トラック）= `{OUTPUT_DIR}\上級編\`
答案を書くトラック。被告人（甲乙丙…）の数を M とすると **(3 + max(1,M)) 〜 本**（047 は M=2 で6本）：

1. `01_問題文.txt` — **初級編 01 と完全同一**（同じ内容をコピー）。
2. 模範答案を**被告人ごとに分割朗読**（`#answer-full` の `model-answer` を HTML 原文どおり）：
   - `02_模範答案a_{被告X罪責}.txt` — 開き「模範答案、その一、{X}の罪責。」
   - `02_模範答案b_{被告Y罪責}.txt` — 開き「模範答案、その二、{Y}の罪責。」（被告が増えれば c, d …）
   - 被告区分が乏しく1本が適切なら `02_模範答案.txt`（開き「模範答案。」）1本でよい。
3. `03_答案構成.txt` — 開き「答案構成。」→ 配点比率（数値・厚く書く重心。当日の分単位時間配分は書かない）＋
   骨組みの通し読み＋書かなくていいことの明示（`#exec-summary`戦略＋`#grading-comment`分量）。
4. `04_差がつく点.txt` — 開き「差がつく点。」→ 上位と中位を分ける点＋陥りやすいミスと減点幅
   （`D.採点者視点`＋`grading-comment`＋第4部試験戦略）＋誤り発見型問いかけ。
5. `05_総まとめ.txt` — 開き「総まとめ。」→ 最重要3点の想起クイズ＋論点抽出シグナルの条件反射クイズ＋
   応用クイズ＋芋づる式関連知識（実体判例・条文・関連論点。出題履歴は書かない）＋一行要約＋
   **やさしい励まし**で締める（音声全体の最後）。

## Section 4: 文体・規則（design-spec 準拠・要点）

- プレーンテキストのみ。SSML/HTML/マークダウン/角括弧マーク一切禁止（design-spec の「プレーンテキスト出力ルール」）。
- 朗読部分（問題文・模範答案・規範コア）は**HTML 原文を忠実に**（要約・順序変更禁止・文体維持・`<a class="xref">`は無視）。
- HTML にない論点・規範・判例の独自追加は禁止（HTML忠実原則）。
- 「君」呼び固定。「皆さん」「あなた」禁止。砕けたタメ口・たとえ話・軽いイジりで飽きさせない。
- アクティブリコール：問いかけを多用（全体で20回以上目安）。ただし問いかけ直後に思考時間の「……」・繋ぎは入れず自然に答えへ。
- 各ファイルは**タイトル読み上げ（例「規範一、事後強盗の身分。」）で開始**。番号ラベル（「48の2」式）は二段構成では使わない。
- 各ファイル概ね 2000〜4200字（design-spec の上限/下限の範囲内）。朗読本体が長い場合もファイル内に収める。
- 条文番号は正式名称、判例は年月日まで、規範引用は文語調そのまま。

## Section 5: 入出力ルール

- `{OUTPUT_DIR}\初級編\` と `{OUTPUT_DIR}\上級編\` を作成（無ければ mkdir）。
- 既存の**フラット連番ファイル（`{PROBLEM_ID}-*.txt`）は触らない**（削除は呼び出し側が生成成功後に行う）。
- 文字コードは UTF-8（BOM なし）。ファイル名は本仕様の形式厳守（2桁ゼロ埋め＋日本語タイトル）。
- 字数計測：`python3 -c "s=open('PATH',encoding='utf-8').read(); import re; print(len(re.sub(r'\s','',s)))"`。

## Section 6: 自己完結（1セッション1完結）

全ファイルを1回の応答内で生成完了させる。ユーザーに確認・予告・打切り文言を一切出さない。
Write を必要数連続実行し、字数調整も連続実行し、最後まで自走する。

## Section 7: 完了 sentinel（全ファイル生成後に必ず出力）

初級編・上級編の全ファイルを書き終えたら、最後に標準出力へ次の1行だけを echo して即終了：

```
BATCH_ITEM_COMPLETED:{PROBLEM_ID}
```

自己検証で疑義が残る場合は代わりに：

```
BATCH_ITEM_COMPLETED_WITH_ISSUES:{PROBLEM_ID}:errors=<N>:warnings=<M>
```

生成不能（HTML 読込不可・致命的不整合）の場合のみ：

```
BATCH_ITEM_FAILED:{PROBLEM_ID}
```
