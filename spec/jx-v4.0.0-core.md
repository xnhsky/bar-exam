## 第0項 四大原理

### 原理1 単一ファイル維持・前半コア／後半 deep（第一原理）

JX は 1 問 = 1 HTML を維持する。TX v11 のような物理2ファイル分割（コア `.html` ＋
別冊 `-deep.html`）は採らない。**1つの HTML 内でセクションの物理順序を再編**し、
前半を**コア層**（周回＋誤答修正で自己完結）、後半を**deep 層**（深掘り）に分ける。

- 前半コア＝第0部凡例 → 第1部分析（exec-summary 削除後）→ 第2部 A〜H → 第3部模範答案・採点講評。
- 後半 deep＝第4部体系化 → 第5部完全プロファイル。
- 学習者は毎周まず前半コアに当たり、後半 deep は解禁条件成立時・直前期に下って読む。

### 原理2 生成方式は (A) 一括生成＋物理順序再編（(B) 後追い追記は却下）

- **(A) 採用：** JX 本体は従来どおり ATHENA から 1 問を**一括鋳造**し、HTML 内の**物理順序
  だけ**をコア前半／deep 後半へ並べ替える。deep を別生成・後追い追記しない。
- **(B) 却下：** コア先行生成＋同一ファイルへ deep 後追い追記は採らない。却下根拠：
  1. **副産物パイプラインが deep に依存**（原理4）。コア完成時点で ARB/TTS が生成できず、
     deep 追記後に再走させる二度手間。(B) の生成経済の利得を相殺する。
  2. **immutable 違反・巨大 diff・二台運用衝突。** 「生成=コミット=immutable」（CLAUDE.md §9）
     運用で、約 250KB 単一ファイルへの後追い追記は元ファイルを必ず書き換え、巨大 diff・
     再生成衝突（§8 二台運用）を踏む。TX が (B 類似) を採れたのは別ファイル（`-deep.html`）
     だったから＝1枚もの維持の JX には移植不可。
  3. **需要が小さい。** JX は 400 問×1〜2 時間で人手照合ガード（CLAUDE.md §4-3 手順0-1）が
     重く、TX のような大量高速回転ではない＝「コア先行で素材を早く配る」需要が小さい。
- **重要な代償（明示）：** (A) は**生成時間の削減がゼロ**。本仕様の効果は学習者の
  **周回エルゴノミクス改善**であって、開発側の生成経済ではない。400 問×1〜2 時間の総量は
  変わらない（生成経済は第11項参照）。

### 原理3 三層モデルとの対応・前倒し防止の主手段は reveal/折りたたみ

| 導き書の層 | v4 LOOP-FOLD での実体 | 使う場面 |
|---|---|---|
| 周回層 | 第1部足場（事案理解）＋第2部 D（解法）/H（チェック）＋RX 論証カード（外部副産物）＋第3部模範答案の reveal 照合 | 毎周 |
| 誤答修正層 | ズレた論点の第2部該当サブ（A/B/C/E）＋`.lecturer-advice`＋採点講評の失点パターン表 | 誤答時のみ |
| 深掘り層 | 後半 deep（第4部体系化＋第5部完全プロファイル）＝デフォルト折りたたみ | 解禁条件成立時と直前期のみ |

単一ファイルゆえ深掘り層が物理的に同居するため、**前倒し防止の主手段は「reveal」と
「折りたたみ」**になる（TX v11 は2ファイル化で折りたたみを不要としたが、JX では再浮上する）。
模範答案・採点講評は reveal で封じ（第4項）、後半 deep は `<details>` でデフォルト折りたたみ
（第4項）する。

### 原理4 副産物（RX/ARB/TTS）は確定済み JX HTML の DOM を素材にする

RX 論証カード・ARB 樹形図・耳トレ TTS 台本は、PDF や逐語ではなく**確定済み JX HTML
（`{SOURCE_HTML_PATH}`）を唯一の素材とするステートレス変換**である（各 headless プロンプト
Section 2-3）。一次情報源は次のとおり：

- **RX**＝第2部のみ（A〜H＋補助で第5部 5-4）
- **ARB**＝第1部（issue-extraction が樹の主題）＋第2部 A〜H＋**第4部**（論点相関・思考フロー）＋ 5-1/5-2
- **TTS**＝第1部＋第2部＋第3部＋**第4部**（試験戦略）

**帰結（折りたたみの鉄則）：** 折りたたみ・順序再編は**素材可用性に影響してはならない**。
`<details>` closed は表示の出し分けであって DOM 除去ではないため、要素を DOM に残す限り
副産物生成・validate-jx（いずれも DOM 走査）は無改修で動く。**deep を物理削除・別生成すると
ARB/TTS が壊れる**ため、「折りたたみ＝DOM 温存」を明文規律とする。

## 第1項 ファイル体系・命名

- JX 本体：`outputs/jx/{科目JX}/{接頭辞}{NNN}.html`（CLAUDE.md §2 と同一・**単一ファイル**）。
- **`-deep.html` は作らない**（TX v11 との違い）。番号抽出・前ゼロ規則は §2-3 を継承。
- 物理構成は「前半コア領域 → 後半 deep 領域」。境界は最初の `<details id="deep-dive">`（第4部冒頭）。
- DOM 温存の鉄則（原理4）：折りたたみ・reveal はいずれも要素を DOM に残す（`display:none`
  での除去や要素削除は禁止。`<details>` closed / reveal 前 hidden で表示だけ抑止する）。

## 第3項 第1部の規律（exec-summary 削除・事案足場の保持）

- **エグゼクティブサマリー（`#exec-summary`・現 ATHENA 約 L1072-1169・約 3.5K）を物理削除する。**
  「答えを先に見ずに答案構成する」導き書原則（reveal 前確定）の最大の漏れ穴であり、
  かつ生成コストも削れる。
- **削除しても復活させない要素：** サマリー内の「解答の骨子（Y/X/共犯の結論先出し）」
  「優先度・トリガーファクト付きの争点完全リスト」＝結論・処理順のネタバレ。
- **必ず残す事案足場：** 本問の事案概要・登場人物図・時系列・ファクト仕分け。1周目の
  事案読込（15 分）の土台であり、答えを示さない。
- **issue-extraction は残すが薄める：** 「処理順・論点数・優先度を明かす表」→
  「**順序・優先度を伏せた論点見出しリスト**（規範・結論を含まない）」へ内容を落とす。
  理由＝ARB が issue-extraction を「中心論点＝樹の主題」抽出の必須一次情報源にするため
  消せない（原理4）が、先読みで論点抽出の思考を奪わないよう値は伏せる。

## 第4項 reveal／折りたたみの規律（前倒し防止の主手段）

### 4-1 後半 deep の折りたたみ
- 第4部＋第5部を 1 個の `<details id="deep-dive">` で包み**デフォルト closed**。
  summary 例＝「📚 深掘り層（第4部体系化＋第5部参考資料）｜1周目は開かない／同じ論点を
  2 回ズラしたら開く／直前期」。
- 開状態は localStorage 保存可（直前期は毎回開いて使うフェーズのため）。
- **xref（本文→`#ref-*` への fragment jump）は closed でも飛べるよう、クリック時に
  `details.open=true` にしてから `scrollIntoView` する JS を付与**（リンク破断ゼロ）。
- **print：** print CSS で `details{ display:block }`／`summary{ display:none }` 相当により
  強制展開（現 ATHENA の print 規則 ~L909-919 を継承・全文が紙に出る）。
- **全文検索フォールバック：** ブラウザ Ctrl+F が closed 要素を拾わない場合に備え、find 時
  展開のフォールバックを検討（第12項）。

### 4-2 模範答案＋採点講評の reveal
- 第3部（`#answer-full` の `.model-answer` ＋ `#grading-comment` の `.grading`）を前半コア末尾に
  置いたまま `<details class="reveal-answer">` でラップ・**既定 closed**。
  summary 例＝「✋ 自分の答案構成を確定してから開く（模範答案＋採点講評）」。
- **localStorage で開閉を保存しない**＝毎周必ず閉じた状態で開始し、「先に見ない」を毎周再強制
  （導き書第4章「模範答案を先に読まない＝reveal 前確定と同原則」）。※保存可否は第12項。
- 採点講評の失点パターン表（規範／あてはめ／結論／文章の4軸）は同じ details 内・照合の最後に開く。

### 4-3 DOM 温存・Lexia 互換
- 折りたたみ・reveal とも要素を DOM に残す（原理4）。`.model-answer::before`「MODEL ANSWER」
  J12・`.grading::before`「GRADING」J13 は details 内でも CSS 検査なので不通過にならない。
- **`<script>...</script>` 内に body 閉じタグ文字列を含めない**（CLAUDE.md §7・Lexia 正規表現対策）。

## 第5項 照合ナビ・口頭骨格（H 兼用）

### 5-1 照合ナビ
- 前半コア内（第3部 reveal の直前）に軽量な「照合ナビ」を新設し、導き書の照合3段を
  既存 A〜H へアンカー対応づける：
  - ① 論点の過不足 → `#issue-extraction`（薄めた見出しリスト）＋各論点の H
  - ② 規範の正確性 → 各論点の G（頻出論証＝規範コア）と A（条文分析）
  - ③ あてはめに使った事実 → 各論点の D（解法アルゴリズム）と F（答案表現集）
- 各 H/G/D/F に id を付与してジャンプ可能に。**part2 の物理順序（A〜H）は変えない。**
- **規律：** 照合ナビは「何をどの順で照合するか」の手順のみ書く。「答えの値」「論点数」は書かない
  （間接ネタバレ防止）。

### 5-2 口頭骨格は H チェックリストを兼用
- 口頭構成 3〜5 分（3周目以降）専用の**新規サマリー・ブロックは作らない**。
- 各論点の H（失点回避チェックリスト・既存4軸＝規範／あてはめ／結論／文章）に、
  「論点名→根拠条文→規範一行→結論」を口頭で言えるかの自己テスト4点を**末尾の小ブロックで**統合。
- **矛盾しない理由：** 口頭骨格は「自分で言えるかをテストする枠（自己生成）」、
  エグゼクティブサマリーは「答えを先に見せる完成済み要約（reveal 破り）」で機能が逆。
  既存 H への統合が最小コスト最大整合（新規ブロックは生成コスト増＋ネタバレ要約復活リスク）。

## 第6項 論証ブロック G の扱い（RX 委譲・※ ATHENA 実装で当初案を撤回）

- **当初案（v0.1・撤回）：** 第2部 G（頻出論証）の【推奨／発展】層を 5-4（deep）へ移し、
  コア G を規範コアに絞って重複を削る。
- **ATHENA 実装で判明（v0.2・2026-06-13）：** 実構造では **G ＝論証本体**（`arg-card`・
  `#arg-Y-1`〜`#arg-X-3`）であり、**コアにある模範答案（reveal 内）が `→ 論証Y-N` で
  G を xref 参照**している。一方 5-4（`#ref-arg`）は各論証の**出典・使用条文・判例・学説の
  学術的付属情報**で、G と重複せず補完関係にある（当初案が前提した「G の3層 ⊃ 5-4の3層の
  重複」は実構造では成立しない）。
- **結論：G 論証本体はコアに残す。** G を deep へ移すと、(1)模範答案の xref が閉じた deep を指し、
  (2)誤答修正で論証の筋を確認するのに deep を開く必要が生じて**コア自己充足（原理3）が崩れる**。
  G は「模範答案の理由づけ参照先」＝**誤答修正層としてコアに属する**。
- **RX 委譲の正しい意味：** RX 論証カードが肩代わりするのは「論証の**反復暗記のための再読**」であって、
  G 論証本体の**配置**ではない。よってコア前半から G を削らない。生成量削減は exec-summary 削除
  （第3項）で達成済みとし、G の重複削減は行わない。
- F（答案表現集）はコアに残す（1周目の構成直後に参照する周回層素材）。

## 第7項 正典（canonical）

- **ATHENA は2分割せず単一正典のまま、セクション順序を「前半コア → 後半 deep」へ再編**する。
  新規執筆ではなく**現 ATHENA の再編**（品質リスク最小化）。
- 再編作業：(1) `#exec-summary` を削除（事案足場は残す・第3項）、(2) 第3部を
  `<details class="reveal-answer">` で包む、(3) 第4部＋第5部を `<details id="deep-dive">` で包み
  後半へ移動、(4) 照合ナビの骨格を追加、(5) H に口頭骨格小ブロックを追加、(6) G を絞り
  3層を 5-4 へ移設、(7) xref open-on-jump JS・print 強制展開 CSS を確認。
- **deep の `<details>` 骨格・reveal 骨格・照合ナビ骨格は構造シェルとして ATHENA に作り込む**
  ＝逐語コピー対象（本文だけ新規執筆）。これでバッチ生成でも折りたたみ・reveal 骨格が
  必ず正典品質で揃う。
- content-independence（構造シェルのみ逐語コピー・本文は完全新規執筆＝CLAUDE.md §4-4）を継承。

## 第8項 検証（validate-jx の core/deep 領域分割）

- `scripts/validate-jx.py` を**1本のまま**、分割点（最初の `#deep-dive` details）で
